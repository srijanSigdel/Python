# Security Policy

## Supported Versions

The following branches and releases receive security updates. Older or inactive versions are not supported.

| Version/Branch | Supported          |
| -------------- | ------------------ |
| main           | :white_check_mark: |
| release-1.x    | :white_check_mark: |
| 0.x            | :x:                |

## Reporting a Vulnerability

If you identify a security vulnerability, follow the process below.

### How to Report
Please report vulnerabilities privately and do not create a public issue.

Contact: security@example.com

Include:
- Description of the issue
- Steps to reproduce
- Potential impact (if known)
- Any logs or proof-of-concept (if applicable)

### Response Timeline
| Action | Timeline |
|--------|----------|
| Acknowledgement | Within 48 hours |
| Initial triage | Within 5 business days |
| Fix plan shared | Within 7 business days |
| Security patch released | Based on severity |

Severity levels follow common scoring standards (e.g., CVSS).

### After Reporting
Once fixed:
- A security advisory may be published
- Credit will be given if disclosure guidelines are followed

## Security Best Practices

To maintain repository security, contributors must follow these requirements:

- Do not commit secrets (API keys, tokens, passwords)
- Use MFA on GitHub accounts
- Submit changes via Pull Requests only
- Ensure all automated security checks pass
- Review dependency alerts and apply updates promptly
